# NLP Trader 日本語版

[English README](README.md)

使ってよいニュースや文章には、株価と出来高だけでは拾えない情報があるのか。
NLP Trader は、その問いを手元のデータで確かめるための研究用ツールです。

検証で厄介なのは、モデルよりもデータの時刻です。たとえば、当時まだ読めなかった記事が
過去の判断に一件混ざるだけで、成績は簡単によく見えます。NLP Trader は各データがいつ利用
できたかを確認し、設定、入力データ、コードの版、結果を実行ごとに残します。Apple Silicon Mac
を含む、手元のパソコンだけで動かせます。

> ここで得られる結果は、すべて仮定に基づく研究上の試算です。投資助言ではなく、利益が出る
> ことを示すものでもありません。研究用コマンドと模擬運用の `paper` コマンドは注文を
> 送りません。別系統の [kabuS コマンド](docs/broker.md) だけは、本番用に設定したうえで
> 明示的に確認して実行すると、実際の株式注文を送ることがあります。

## まず動かす

Python 3.12 と [`uv`](https://docs.astral.sh/uv/) があれば動かせます。

```bash
uv sync --locked
uv run nlp-trader validate-config --config configs/sample.yaml
uv run nlp-trader smoke --config configs/sample.yaml
```

サンプルが使うのは、リポジトリに同梱した小さな架空データです。市場データサービスの契約も、
認証情報も、GPU も、ダウンロード済みの言語モデルも要りません。

実行が終わると、`run_id` という実行番号と保存先が表示されます。まずは次の三つを確認します。

- `reports/sample/<run_id>/research_note.md` — 結果と限界を読みやすくまとめた記録
- `reports/sample/<run_id>/run.final.json` — 設定、入力のハッシュ、コードの版、出力一覧
- `data/processed/sample/<run_id>/backtests/` — 模擬運用での保有状況、売買、費用の詳細

この数値から分かるのは、処理が最後まで動いたことだけです。将来の収益性は判断できません。

## 何を検証するのか

一度の実行で、NLP Trader は次の処理を行います。

1. 市場データと文章データを読み込み、元のファイルは変更しない
2. 過去の各判断に、その時点で利用できた情報だけを渡す
3. モデルへの入力と、あとで分かる値動きを別々に作る
4. 過去の期間で学習し、その次の期間を評価する手順を繰り返す
5. 売買の開始と終了、費用、保有量や売買量の上限を模擬する
6. 人向けの報告書と、機械で再検証できる実行記録を保存する

日次シミュレーションでは、必要なデータがそろった後に判断します。模擬売買を始めるのは、
早くても次に市場が開く時点です。設定した取引日数が過ぎたら終了します。どの価格を使い、
費用をどう計算するかは、[バックテスト仕様](docs/backtesting.md) に記載しています。

比べるのは、価格・出来高だけを使う方法、文章だけを使う方法、両方を使う方法だけでは
ありません。全銘柄を同じ比率で持つ方法、直近の値動きだけを見る方法、何も売買しない方法も、
同じ実行結果に残します。文章データを加える意味があるかどうかは、こうした単純な比較対象を
上回って初めて判断できます。

## 設定を選ぶ

| 設定ファイル | 用途 | データ |
|---|---|---|
| `configs/sample.yaml` | 最小構成での一通りの動作確認 | 同梱の架空データ |
| `configs/backtest.yaml` | 同梱データを使った、より厳しい動作確認 | 同梱の架空データ |
| `configs/local.yaml` | 手元のファイルを使う研究 | 利用権を確認したローカルデータ |
| `configs/japan_baseline.yaml` | 日本株の研究 | 利用権を確認した日本市場データ |

`local.yaml` と `japan_baseline.yaml` は、入力ファイルを用意するまで検証に失敗します。
データ提供元のファイルは同梱しておらず、自動取得もしません。受け付ける形式は
[入力データ](docs/input_data.md)、日本株向けの条件は [日本株ベースライン](docs/japan_baseline.md)
で確認できます。

## 手元のデータを使う

1. `configs/local.yaml` または `configs/japan_baseline.yaml` をコピーする
2. 銘柄、市場データ、文章データのファイルパスを指定する
3. 仮の利用許諾情報を、自分のデータ利用権を確認できる記録に置き換える
4. 手元のメモリに収まる日付範囲と銘柄を選ぶ
5. バックテストの前に設定を検証する

```bash
uv run nlp-trader validate-config --config configs/local.yaml
uv run nlp-trader backtest --config configs/local.yaml --limit 100
```

開発中の確認なら、日付や銘柄を絞ると短時間で終わります。

```bash
uv run nlp-trader backtest \
  --config configs/local.yaml \
  --start-date 2024-01-01 \
  --end-date 2025-12-31 \
  --symbol AAPL \
  --limit 100
```

研究用コマンドは、実行のたびに新しいディレクトリを作ります。以前の実行を上書きせず、
途中からの継ぎ足しもしません。コマンドの一覧と、それぞれが事前に行う処理は
[ワークフロー](docs/workflows.md) で確認できます。

入力データ、生成物、学習済みファイルは、通常 Git の管理対象外です。認証情報、契約データ、
非公開の文章データを、ソースコードと一緒に追加しないでください。

### 日本株データを用意する

既定の日本株設定は [`configs/japan_baseline.yaml`](configs/japan_baseline.yaml) です。
利用権を自分で確認した入力ファイルを、次の場所に置きます。

```text
data/local/japan/assets.csv
data/local/japan/market_bars.parquet
data/local/japan/text_items.jsonl
```

- `assets.csv` — 銘柄コード、名称、業種、売買単位など
- `market_bars.parquet` — 日ごとの始値、高値、安値、終値、売買量など
- `text_items.jsonl` — 利用を許されたニュースや文章。市場データだけを試す場合は空でもよい

空の文章ファイルが必要な場合も、既存ファイルを空にして作り直してはいけません。ファイルが
まだ存在しないことを確認してから作成します。

```bash
mkdir -p data/local/japan
touch data/local/japan/text_items.jsonl
wc -c data/local/japan/text_items.jsonl
```

最後の表示が `0` なら空のファイルです。準備ができたら、設定の検証、取り込み、バックテストの
順に実行します。

```bash
uv run nlp-trader validate-config --config configs/japan_baseline.yaml
uv run nlp-trader ingest-market --config configs/japan_baseline.yaml
uv run nlp-trader backtest --config configs/japan_baseline.yaml --limit 320
```

必要なファイルがなければ、最初の検証で明示的に失敗します。列、時刻、取引日の条件も厳密です。
株式分割などがある場合でも、模擬売買には入力された当日の実価格を使います。収益率の調整に
使えるのは、その時点ですでに利用できた係数だけです。現在から過去を振り返って作った調整済み
価格を、過去の売買価格として使うことはありません。

## ローカル言語モデルを追加する

大規模言語モデル（LLM）による文章分析は、標準では使いません。通常のサンプルと基準モデルは、
LLM 用ライブラリも、モデルのダウンロードも、PyTorch も必要としません。

任意機能を使うには、追加依存グループ `llm` に含まれる `llama-cpp-python==0.3.34` を
インストールします。読み込むのは、設定したパスにある一つの GGUF ファイルだけです。GGUF は
llama.cpp が使うモデル形式で、処理中にモデルをダウンロードすることはありません。

同梱設定では `unsloth/Qwen3.6-27B-MTP-GGUF:UD-Q4_K_XL` を、リビジョン
`5c641ee6f93ccf8b1f01824455bfdbbdd7d658bf` に固定しています。想定するローカルファイル名は
`Qwen3.6-27B-UD-Q4_K_XL.gguf` です。

このファイルだけで約 17.9 GB あります。実用上は、32 GB 以上のユニファイドメモリを持つ Mac で、
文章を一件ずつ処理してください。推論にはモデル本体以外のメモリも必要です。利用条件を確認して
自分でファイルを取得し、`paths.llm_model` にそのファイルのパスを直接指定します。

モデルが返すのは、プログラムが文章に結び付けた会社ごとの構造化データです。文章が示す方向、
確信度、想定期間、根拠となる文章、起こり得る出来事、判断が外れる理由が含まれます。
プログラムは、出力形式と時刻、すべての対象会社について結果があるか、根拠 ID が正しいかを
検査します。説明に数値があれば、引用箇所に同じ数値があるかも確かめます。

検査できるのは、そこまでです。情報源が正しいか、モデルの解釈が妥当かまでは証明できません。

LLM の値は、通常の文章スコアとは分けて保存します。そのため、市場データ、通常の文章分析、
LLM 分析、それらの組み合わせを同じ条件で比較できます。モデルが返した確信度も独立した入力
として残し、検証済みの確率や保有量とはみなしません。

LLM に渡すのは、対象の文章から切り出した箇所、結び付けた会社、情報源の種類と品質スコア、
予測期間です。価格、将来の値動き、模擬運用中の保有状況、ウェブ検索結果、別の文書は渡さず、注文も
作らせません。ただし、学習済みモデルが、その判断時点より後の事実をすでに知っている可能性は
残ります。この検証だけで「当時のモデルも同じ出力をした」とは言えません。

モデル名には `MTP` が含まれますが、現在の Python 実装はその追加高速化を使いません。通常の
自動テストでは、モデルの代わりに固定応答を使って周辺処理を検証します。手元の実ファイルを
読み込んで仕様どおりの応答を返せるかどうかは、別の実モデル受け入れテストで確かめます。導入、取得、
ハッシュ確認、設定、実モデルテストの手順は、
[ワークフロー](docs/workflows.md#optional-local-generative-semanticevidence-annotation) にあります。

## ローカル研究エージェントを使う

研究エージェント機能も、標準では無効です。ローカルの分析役に見せるのは、開発期間だけを含む
封印済みのデータ一式です。使えるツールは四つで、どれも読み取り専用です。分析役は根拠を
調べて提案を出すか、根拠が足りなければ提案を見送ります。同じ入力に同じ判定を返す検証器が、
参照先、比較対象、時点の整合性、禁止された操作を確認します。

分析役には、実験を動かす権限がありません。最終評価期間を開くことも、模擬注文案を作る
ことも、実際の注文を出すこともできません。

`nlp-trader agent-study ...` は、人が操作する管理用コマンドです。研究の作成と予約、封印ビューの
出力、検証済み提案のコンパイル、人による開発実行の承認、開発期間だけを使った比較、候補の
凍結、登録済み最終評価期間の一度限りの開示、系譜の監査を処理します。モデルを読み込む
`propose` と、モデルを読み込まずに記録を確かめる `verify`、`replay` は、別の
`nlp-trader-agent` 実行ファイルに分かれています。この機能を使わない限り、通常のパイプラインは
変わりません。

```bash
uv run nlp-trader agent-study --help
uv run nlp-trader-agent --help
```

これは自律売買のための機能ではありません。ローカル研究で使える権限を、必要な範囲に閉じ込める
仕組みです。開発実行には、実行内容と完全に一致する承認記録が必要です。一度凍結した候補は
元に戻せず、最終評価期間の重複利用はシステム全体で記録されます。出力は最後まで仮定に基づく
研究結果です。
詳しくは [ワークフロー](docs/workflows.md#local-research-agent-study) と
[研究手順](docs/research_protocol.md#local-agent-study-protocol) を参照してください。

## 先に知っておくべき限界

- 過去の成績が良くても、将来も同じ成績になるとは限りません。
- 手数料、買値と売値の差、売買に伴う価格変化、空売り費用、実行可能な売買量はいずれも推定です。
  取引所や証券会社の動きを完全に再現するものではありません。
- 最終評価期間を見た後で戦略を繰り返し変えると、その評価は信用できなくなります。
- 入力データには、どの銘柄が当時存在したか、どの版の報告書が使えたか、各値がいつ届いたかが
  正しく記録されている必要があります。
- 大きな実行では、途中結果の表も最終段階でメモリに載ります。最初は日付、銘柄、`--limit` で
  範囲を絞ってください。
- 任意の言語モデル機能が使うのは、ローカルのモデルファイルだけです。通常のサンプルに
  PyTorch や `llama-cpp-python` は要りません。
- ウェブサイトからデータを収集する機能も、市場データをダウンロードする機能もありません。

## 実注文コマンドは別系統

研究パイプラインは証券会社へ接続しません。kabuS コマンドは所有者本人だけが使う別系統の機能で、
kabuStation が動いている Windows パソコン上で明示的に実行します。

```bash
uv run nlp-trader broker --help
```

検証環境が返すのは固定されたテスト応答だけです。実際の注文が送られるのは、本番用の設定で
`submit-order` を明示的に実行し、確認ダイジェストと `REAL_ORDERS` の本番確認を渡した場合に
限られます。

Codex を補助に使って開発したコードも、所有者自身のプログラムとして扱います。ただし、認証情報や
操作手段を他人へ渡してはいけません。接続先、注文上限、停止方法、確認手順を含む
[kabuS 運用ガイド](docs/broker.md) を、本番用に設定する前に必ず確認してください。

サンプルを動かして分かるのは、プログラムが最後まで動くことだけです。ニュースや文章に追加の
価値があるかは、利用権を確認したデータを入れ、単純な基準と同じ条件で比べて、初めて検討できます。
実注文は、その研究とは別です。
