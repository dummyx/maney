# NLP Trader 日本語版

[English README](README.md)

利用を許されたニュースや文章は、株価と出来高だけでは見えない情報を持っているのか。
NLP Trader は、その一点を手元のデータで繰り返し検証するための研究システムです。

ただし、過去の判断に当時まだ届いていなかった記事を一つ混ぜるだけで、結果は簡単に
よく見えます。そこで、各データが実際に利用可能になった時刻を守り、設定、入力、
コードの版、結果を実行ごとに記録します。動かす場所は、Apple Silicon Mac を含む
ローカルのパソコンです。

> すべての結果は仮定に基づく研究上の試算です。投資助言ではなく、利益を示すものでも
> ありません。研究用コマンドと模擬運用の `paper` コマンドは注文を送りません。別系統の
> [kabuS コマンド](docs/broker.md) だけは、明示的に本番設定と確認を行った場合に実際の
> 株式注文を送ることがあります。

## まず動かす

必要なのは Python 3.12 と [`uv`](https://docs.astral.sh/uv/) です。

```bash
uv sync --locked
uv run nlp-trader validate-config --config configs/sample.yaml
uv run nlp-trader smoke --config configs/sample.yaml
```

見本では、リポジトリに同梱した小さな架空データを使います。市場データの契約、認証情報、
GPU、ダウンロード済みの言語モデルは要りません。

実行が終わると、`run_id` という実行番号と保存先が表示されます。最初に確かめるのは、
次の三つです。

- `reports/sample/<run_id>/research_note.md` — 結果と限界を人が読める形でまとめた記録
- `reports/sample/<run_id>/run.final.json` — 設定、入力のハッシュ、コードの版、出力一覧
- `data/processed/sample/<run_id>/backtests/` — 模擬の保有、売買、費用の詳細

見本の数値が証明するのは、処理が最後まで動いたことだけです。将来の収益とは結び付きません。

## 何を検証するのか

一回の実行で、NLP Trader は次の処理を行います。

1. 市場データと文章データを読み込み、元のファイルは変更しない
2. 過去の各判断に、その時点で利用できた情報だけを渡す
3. モデルへ渡す特徴と、あとで判明する値動きを別々に作る
4. 過去の期間で学習し、その次の期間を評価する手順を繰り返す
5. 売買の開始と終了、費用、保有量や売買量の上限を模擬する
6. 人向けの報告書と、機械で再検証できる実行記録を保存する

現在の日次シミュレーションは、必要なデータがそろってから判断します。模擬売買は早くても
次の市場開始時に始まり、設定した取引日数の経過後に終わります。使う価格と費用の規則は
[バックテスト仕様](docs/backtesting.md) に記載しています。

比較するのは、価格・出来高だけを使う方法、文章だけを使う方法、両方を使う方法だけでは
ありません。全銘柄を同じ比率で持つ方法、直近の値動きだけを見る方法、何も売買しない方法も
同じ実行に残します。文章を加えた価値は、こうした素朴な基準を越えて初めて見えてきます。

## 設定を選ぶ

| 設定ファイル | 用途 | データ |
|---|---|---|
| `configs/sample.yaml` | 最短の一貫動作確認 | 同梱の架空データ |
| `configs/backtest.yaml` | 同梱データを使った、より厳しい確認 | 同梱の架空データ |
| `configs/local.yaml` | 手元のファイルを使う研究 | 自分が利用を許されているローカルデータ |
| `configs/japan_baseline.yaml` | 日本株の研究 | 自分が利用を許されている日本市場データ |

`local.yaml` と `japan_baseline.yaml` は、入力ファイルを用意するまで検証に失敗します。
プロジェクトは販売元のデータを同梱せず、自動取得もしません。受け付ける形式は
[入力データ](docs/input_data.md)、日本株向けの条件は
[日本株ベースライン](docs/japan_baseline.md) で確認できます。

## 手元のデータを使う

1. `configs/local.yaml` または `configs/japan_baseline.yaml` をコピーする
2. 銘柄、市場、文章の各ファイルを設定する
3. 仮の利用許諾欄を、自分のデータ利用権を確認できる記録に置き換える
4. ローカルメモリに収まる日付範囲と銘柄を選ぶ
5. バックテストの前に設定を検証する

```bash
uv run nlp-trader validate-config --config configs/local.yaml
uv run nlp-trader backtest --config configs/local.yaml --limit 100
```

開発中は、日付や銘柄を絞ると早く確かめられます。

```bash
uv run nlp-trader backtest \
  --config configs/local.yaml \
  --start-date 2024-01-01 \
  --end-date 2025-12-31 \
  --symbol AAPL \
  --limit 100
```

研究用コマンドは実行のたびに新しいディレクトリを作り、以前の実行を上書きしたり、
途中から継ぎ足したりしません。コマンドの一覧と、各コマンドが先に行う処理は
[ワークフロー](docs/workflows.md) にあります。

入力、生成した成果物、学習済みファイルは通常 Git の管理対象外です。認証情報、契約データ、
非公開の文章データをソースコードと一緒に追加しないでください。

### 日本株データを用意する

既定の日本株設定は [`configs/japan_baseline.yaml`](configs/japan_baseline.yaml) です。
入力ファイルは自分で利用権を確認し、次の場所に置きます。

```text
data/local/japan/assets.csv
data/local/japan/market_bars.parquet
data/local/japan/text_items.jsonl
```

- `assets.csv` — 銘柄コード、名称、業種、売買単位など
- `market_bars.parquet` — 日ごとの始値、高値、安値、終値、売買量など
- `text_items.jsonl` — 利用を許されたニュースや文章。市場データだけを試す場合は空でもよい

空の文章ファイルが必要でも、既存のファイルを消してはいけません。まだ存在しないことを
確認してから作成します。

```bash
mkdir -p data/local/japan
touch data/local/japan/text_items.jsonl
wc -c data/local/japan/text_items.jsonl
```

最後の表示が `0` なら空です。準備ができたら、設定、取り込み、バックテストの順に
実行できます。

```bash
uv run nlp-trader validate-config --config configs/japan_baseline.yaml
uv run nlp-trader ingest-market --config configs/japan_baseline.yaml
uv run nlp-trader backtest --config configs/japan_baseline.yaml --limit 320
```

必要なファイルがなければ、最初の検証は明示的に失敗します。列、時刻、取引日の条件は
厳密です。とくに株式分割などについて、模擬売買には入力された当日の実価格を使い、
収益率の調整には当時すでに利用できた係数だけを使います。現在から過去を振り返って作った
調整済み価格を、過去の売買価格にはしません。

## ローカル言語モデルを追加する

大規模言語モデル（LLM）による文章分析は、初期設定では無効です。通常の見本と基準モデルは
LLM 用ライブラリをインストールせず、モデルをダウンロードせず、PyTorch なしで動きます。

任意機能を使う場合は、別の `llm` 追加依存として `llama-cpp-python==0.3.34` を導入し、
設定したパスにある一つの GGUF ファイルを直接読みます。GGUF は llama.cpp 用のモデル形式です。
処理中にモデルを取得することはありません。

同梱設定が指定するモデルは
`unsloth/Qwen3.6-27B-MTP-GGUF:UD-Q4_K_XL`、固定リビジョンは
`5c641ee6f93ccf8b1f01824455bfdbbdd7d658bf`、ローカルのファイル名は
`Qwen3.6-27B-UD-Q4_K_XL.gguf` です。約 17.9 GB あるため、実用上は 32 GB 以上の
ユニファイドメモリを持つ Mac で、文章を一件ずつ処理してください。モデル以外にもメモリを
使います。利用条件を確認して自分で取得し、`paths.llm_model` に直接のファイルパスを設定します。

モデルは一つの文章から、プログラムが結び付けた会社ごとに、方向、確かさ、想定期間、根拠の
文章、起こり得る出来事、判断が外れる理由を構造化して返します。プログラムは形式、時刻、
対象会社の欠落、根拠 ID を検証し、説明中の数値が引用箇所に存在するかも確かめます。

それでも、情報源の正しさやモデルの解釈までは証明できません。

LLM の値は通常の文章スコアと分けて保存するため、市場データ、通常の文章分析、LLM 分析、
それらの組み合わせを比較できます。モデル自身が出す確信度も別の入力として残し、検証済みの
確率や保有量としては扱いません。

LLM には価格、将来の値動き、現在の模擬保有、ウェブ検索結果、別の文書を渡さず、注文も
作らせません。ただし、現代のモデルは過去の判断時点より後の事実を学習済みかもしれません。
したがって、この検証から「当時のモデルなら同じ出力をした」とは結論できません。

モデル名には `MTP` が含まれますが、現在の Python 実装はその追加高速化を使いません。
通常の自動テストは固定応答で周辺処理を検証します。手元の実ファイルを読み、正しい応答を
生成できるかは、別の実モデル受け入れテストで確かめます。導入、取得、ハッシュ確認、設定、
実モデルテストの手順は
[ワークフロー](docs/workflows.md#optional-local-generative-semanticevidence-annotation) にあります。

## ローカル研究エージェントを使う

研究エージェント機能も初期設定では無効です。ローカルの分析役が見られるのは、開発期間だけを
封印したデータ一式です。四つの読み取り専用ツールで根拠を調べ、提案するか、根拠不足として
提案を見送ります。同じ入力には同じ判定を返す検証器が、参照先、比較対象、時点の整合性、
禁止された操作を確認します。

分析役に実験を動かす権限はありません。最終評価期間を開くことも、模擬注文の意図を作ることも、
実際の注文を出すこともできません。

権限を持つ `nlp-trader agent-study ...` コマンドは、研究の作成と予約、封印ビューの出力、
検証済み提案のコンパイル、人による開発実行の承認、開発期間だけの比較、候補の凍結、登録済み
最終評価期間の一度限りの開示、系譜の監査を処理します。モデルを読み込む `propose` と、
モデルを読み込まずに記録を確かめる `verify`、`replay` は、別の `nlp-trader-agent` 実行
ファイルに分離されています。この機能を使わなければ、通常のパイプラインは変わりません。

```bash
uv run nlp-trader agent-study --help
uv run nlp-trader-agent --help
```

これは自律売買ではなく、ローカル研究の権限を狭く保つ仕組みです。開発実行には内容が完全に
一致する人の承認が必要で、候補の凍結は取り消せず、最終評価期間の重複利用は全体で記録されます。
出力は最後まで仮説上の研究結果です。詳しくは
[ワークフロー](docs/workflows.md#local-research-agent-study) と
[研究手順](docs/research_protocol.md#local-agent-study-protocol) を参照してください。

## 先に知っておくべき限界

- 過去の成績が良くても、将来の成績は分かりません。
- 手数料、買値と売値の差、売買に伴う価格変化、空売り費用、実行可能な売買量は推定です。
  取引所や証券会社を完全には再現しません。
- 最終評価期間を見てから戦略を繰り返し変えると、その評価は信用できなくなります。
- どの銘柄が当時存在したか、どの版の報告書が使えたか、各値がいつ届いたかを、入力データが
  正しく表している必要があります。
- 大きな実行では、途中の表を最終的にメモリへ読み込みます。最初は日付、銘柄、`--limit` で
  範囲を絞ってください。
- 任意の言語モデル機能はローカルファイルだけを使います。通常の見本に PyTorch や
  `llama-cpp-python` は不要です。
- ウェブサイトの収集機能や、市場データのダウンローダーはありません。

## 実注文コマンドは別系統

研究パイプラインは証券会社へ接続しません。kabuS コマンドは所有者本人が別に起動する機能で、
kabuStation が動く Windows パソコン上で明示的に実行します。

```bash
uv run nlp-trader broker --help
```

検証用の設定は固定されたテスト応答を返すだけです。本番設定では実際の注文を送れます。
Codex を補助に使って開発したコードも所有者自身のプログラムとして扱いますが、認証情報や
操作手段を他人へ渡してはいけません。接続先、注文上限、停止方法、確認手順を含む
[kabuS 運用ガイド](docs/broker.md) を、本番設定の前に必ず確認してください。
